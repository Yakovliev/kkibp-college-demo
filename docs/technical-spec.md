# Технічне завдання: демоверсія сайту коледжу

## Мета документа
Документ фіксує прийняті вимоги до демоверсії сайту Економіко-правового фахового коледжу Київського кооперативного інституту бізнесу і права, які потрібно врахувати під час майбутньої WordPress або production-реалізації.

Цей документ не описує всі можливі функції сайту. Він фіксує прийняті UX, адаптивні та технічні вимоги, які вже узгоджені в демоверсії.

Окремі функції можуть бути додані до документа як **кандидати на затвердження**, якщо їх потрібно описати для обговорення майбутньої реалізації. Такі секції мають явний статус і не вважаються фінальними вимогами, доки не буде окремого рішення.

## Контекст проєкту
- Поточна демоверсія є статичним HTML/CSS/JS сайтом.
- Демоверсія публікується через GitHub Pages, тому у репозиторії мають залишатися готові статичні `.html` файли.
- Повторювані глобальні елементи, зокрема header і footer, у статичній демоверсії можуть фізично дублюватися в готових HTML-файлах, але мають синхронізуватися з одного локального джерела, а не редагуватися вручну як незалежні копії.
- Українська версія є основною. Англійська підтримується як часткова мовна версія: англомовний інтерфейс і погоджені статичні сторінки без обов'язкового дублювання всього контенту.
- Майбутня реалізація має зберегти структуру, поведінку та адаптивні нюанси, прийняті в демоверсії.

## Мовна модель сайту
- Основна мова сайту - українська.
- Англійська версія не є повною паралельною копією всього сайту. Вона має покривати глобальний інтерфейс, службові елементи та окремі статичні сторінки.
- Англійською мають бути доступні: header, footer, головна навігація, перемикач мови, пошук, основні кнопки, CTA, службові повідомлення, сторінка 404, глобальні підписи й інші елементи інтерфейсу теми.
- Статичний контент головної сторінки має мати англійську версію. Блоки новин або оголошень на англійській головній сторінці можуть показувати українські новини без перекладу.
- Погоджені статичні сторінки розділів мають мати англійську версію: «Коледж», «Абітурієнту», «Студенту», «Випускнику», «Наука», «Бібліотека» та інші окремо погоджені сторінки.
- Динамічний контент, який регулярно додається через WordPress, зокрема новини, оголошення, документи, сторінки з матеріалами розділів і медіа, за замовчуванням публікується один раз українською мовою.
- Для такого динамічного контенту не потрібно створювати обов'язковий англійський дубль запису, сторінки або медіафайлу.
- На англомовних сторінках, які ведуть до динамічного українського контенту, достатньо перекладати глобальний header, footer і службові елементи інтерфейсу. Сам контент може залишатися українською.
- Перемикач мови зберігає короткі позначення «UA» і «EN» без додаткового пояснювального тексту в інтерфейсі.
- Якщо для конкретної сторінки є погоджена англійська версія, перемикач має вести на неї. Якщо англійської версії немає, не створювати порожню або машинно скопійовану сторінку лише заради формальної двомовності.

## Соціальні прев’ю сторінок
### Прийнята вимога
- Усі публічні сторінки української та англійської версій мають містити метадані для коректного прев’ю в месенджерах і соціальних мережах: стандартний `meta name="description"`, Open Graph-теги і Twitter/X card-теги.
- Логотип коледжу має відображатися у прев’ю з усіх сторінок сайту. Це прийнята вимога для демоверсії і майбутньої production-реалізації.
- Для соціального прев’ю використовується файл `assets/logo_small.gif`. У статичній демоверсії для `og:image` і `twitter:image` потрібно вказувати абсолютний публічний URL GitHub Pages: `https://yakovliev.github.io/kkibp-college-demo/assets/logo_small.gif`.
- Для української версії поточний загальний опис прев’ю: «Офіційний сайт коледжу: вступ, навчання, студентське життя, новини та корисні сервіси.»
- Для англійської версії поточний загальний опис прев’ю: «Official college website: admissions, studies, student life, news and useful services.»
- `og:title` може бути коротшим за браузерний `<title>`, щоб прев’ю в Telegram та інших месенджерах не виглядало перевантаженим повною юридичною назвою.
- `og:url` має відповідати канонічній публічній адресі сторінки на GitHub Pages або production-домені після перенесення.
- `og:image:alt` і `twitter:image:alt` мають бути локалізовані відповідно до мови сторінки.

### Точка для майбутнього затвердження
- **Потребує редакційного затвердження.** Поточна демоверсія використовує однаковий опис соціального прев’ю для всіх сторінок у межах мовної версії.
- Перед production-публікацією потрібно окремо вирішити, чи лишається єдиний опис для всіх сторінок, чи окремі сторінки отримують власні описи.
- Базова рекомендація на поточному етапі: залишити однаковий опис для всіх сторінок, щоб прев’ю було стабільним і не створювало редакційного навантаження. Якщо окремі сторінки стануть важливими для промо або пошуку, можна затвердити сторінкові описи.
- Навіть якщо текст опису буде змінено або деталізовано по сторінках, вимога показувати логотип у прев’ю з усіх сторінок не змінюється.

## Header і головна навігація
### Принцип глобального header
- Header є глобальним елементом сайту і має бути однаковим для всіх сторінок у межах кожної мовної версії.
- Допустимі відмінності між сторінками: активний стан поточної сторінки, мовні тексти, відносні шляхи до ресурсів і мовних версій.
- Українська структура меню є основною структурою сайту. Назви пунктів, груп і підпунктів потрібно відтворювати точно так, як вони зафіксовані нижче.
- URL окремих пунктів можуть уточнюватися під час наповнення сайту, але ієрархія, назви і порядок пунктів не повинні змінюватися без окремого рішення.
- Header не повинен втрачати пошук, перемикач мови, CTA «Вступ» або мобільну навігацію на окремих сторінках.

### Брендовий блок header
- Логотип у header: `assets/logo_small.gif` для українських сторінок і `../assets/logo_small.gif` для англійських сторінок.
- Логотип є декоративним, якщо поруч є текстова назва коледжу, тому може мати порожній `alt`.
- Брендовий блок є посиланням на головну сторінку мовної версії.
- Текст бренду українською:
  - «Економіко-правовий фаховий коледж».
  - «Київського кооперативного інституту бізнесу і права».
- Текст бренду англійською:
  - «Professional College of Economics and Law».
  - «of the Kyiv Cooperative Institute of Business and Law».
- Назву коледжу не скорочувати в основному брендовому блоці header.
- На desktop і tablet назва може відображатися у два логічні рядки через окремі частини назви. На mobile текст має переноситися природно без обрізання.

- На desktop і tablet службові кнопки верхнього header-рядка мають бути уніфіковані за розміром. Кнопки пошуку, перемикання теми і перемикання мови мають мати однакову квадратну площу натискання в межах одного header-рядка.

### Пошук у header
- Пошук відкривається з іконки пошуку в header.
- Search dialog має мати заголовок «Пошук» і текстовий заголовок «Що ви шукаєте?» в українській версії.
- Поле пошуку в українській версії має placeholder «Наприклад, розклад або правила прийому».
- Підказки популярного пошуку в українській версії: «правила прийому», «розклад занять», «електронний каталог».
- Dialog має закриватися кнопкою закриття, Escape і без втрати доступності клавіатурної навігації.

### Поведінка menu і mega-menu
- Клік або tap по пункту з підменю відкриває відповідне mega-menu без зміни URL.
- Повторний клік по відкритому пункту закриває mega-menu.
- Клік по іншому пункту з підменю закриває попереднє меню і відкриває нове.
- Hover не використовується як основний спосіб відкриття mega-menu. Меню має відкриватися через явну дію користувача: click, tap або клавіатурну взаємодію.
- Відкрите mega-menu має закриватися при кліку поза меню, при виборі посилання всередині меню або через Escape.
- Якщо сторінка вже проскролена, відкритий drawer або mega-menu має з'являтися в межах поточного viewport. Користувач не повинен повертатися на початок сторінки, щоб побачити відкрите меню.
- У desktop/tablet header при скролі має залишатися доступним компактний навігаційний ряд, зокрема CTA «Вступ».

### Мобільний header і drawer
- На mobile основна навігація відкривається через кнопку меню в header.
- Кнопка меню має показувати іконку відкриття, іконку закриття, `aria-expanded` і зрозумілий `aria-label`.
- У верхній частині відкритого мобільного меню показується блок «Мова сайту» з опціями «UA» і «EN».
- Пункти з mega-menu у mobile працюють як accordion.
- CTA «Вступ» у mobile drawer має залишатися виділеною синьою кнопкою.
- Відкрите мобільне меню має працювати як drawer: це окремий шар навігації поверх сторінки.

### Скрол мобільного меню
- Коли мобільне меню відкрите, фонова сторінка не повинна скролитися.
- Якщо користувач хоче скролити сторінку, він має спочатку закрити мобільне меню.
- Якщо розкриті підрозділи меню не вміщуються у видиму область, скролиться саме меню.
- Жест скролу всередині drawer не повинен випадково передаватися фоновій сторінці.
- Відкрите меню має зберігати позицію сторінки, з якої його відкрили.
- Відкриття drawer після скролу сторінки не повинно зміщувати header або сам drawer за межі видимої області.
- Плавна accordion-анімація відкриття підрозділів не повинна блокувати внутрішній скрол drawer. Після завершення анімації довгий відкритий підрозділ має залишатися частиною скрольованого мобільного меню, а не окремим “глухим” контейнером.

### Висота мобільного меню
- Відкрите мобільне меню має займати природну висоту свого контенту, коли список короткий.
- Не допускається великий порожній білий простір під останнім пунктом меню.
- Для довгого вмісту drawer має мати максимальну висоту, обмежену доступною висотою екрана, і власний внутрішній скрол.

### Планшетний header і mega-menu
- На tablet-ширинах навігація залишається горизонтальною і може прокручуватися по горизонталі, якщо не вміщується в один ряд.
- На tablet відкриті mega-menu мають працювати як окремий шар поверх сторінки.
- Коли на tablet відкрито mega-menu, фонова сторінка не повинна скролитися.
- Якщо вміст mega-menu не вміщується у видиму область, скролиться саме відкрите меню.
- Жест скролу всередині mega-menu не повинен випадково передаватися фоновій сторінці.
- Відкриття mega-menu після скролу сторінки не повинно зміщувати header або меню вище поточного viewport.
- Довгі розділи, зокрема «Бібліотека», мають адаптивно переносити групи меню на наступні рядки. На tablet не допускається примусове стискання всіх груп в один ряд, якщо це спричиняє накладання або важке читання тексту.
- Текст пунктів меню має переноситися або займати достатню ширину, щоб не наїжджати на сусідні колонки.

## Тема оформлення (день / ніч)
- Пов’язане дизайн-рішення: `docs/decisions/DDR-012-day-night-theme-toggle.md`.
- Сайт підтримує дві теми: основну (день) і нічну (ніч). Багатотемний демо-перемикач і додаткові експериментальні палітри (фіолетова, бордова) у прийнятому дизайні не використовуються.
- Перемикання теми виконується кнопкою в header з іконкою сонця/місяця. Кнопка доступна на desktop, tablet і mobile.
- Іконка показує поточний стан: місяць у денному режимі (клік вмикає ніч), сонце в нічному режимі (клік вмикає день). `aria-label` і `title` кнопки локалізовані й описують дію.
- Вибір теми зберігається між відвідуваннями (у демоверсії - локальне сховище, ключ `site-theme`). Невідоме або застаріле збережене значення трактується як день.
- Нічна тема застосовується через атрибут `data-theme="night"` на кореневому елементі та перевизначає системні CSS-змінні теми. Окремі компоненти не задають власних кольорів під ніч, а використовують спільні змінні.
- У майбутній WordPress-реалізації перемикач теми має бути частиною header-теми, а збереження вибору - через локальне сховище або погоджений механізм теми.

## Головна сторінка
### Hero першого екрана
- Перший екран головної сторінки має бути компактним і не створювати враження надмірного порожнього простору між header, текстовим повідомленням, логотипом і наступним блоком.
- Hero має використовувати офіційний логотип `assets/college_logo_official.gif` як основний візуальний елемент поруч із повідомленням «Освіта, що відкриває можливості».
- Не додавати поруч із великим логотипом окремий текстовий дубль назви коледжу, оскільки назва вже присутня у самому зображенні та в header.
- Зображення великого логотипа має показуватися без зайвого прозорого поля файлу. Візуально потрібно приховувати приблизно 10% висоти зверху і приблизно 30% висоти знизу.
- Обрізання логотипа може бути реалізоване через CSS-вікно з `overflow: hidden` або через окремий підготовлений asset, але результат має відповідати погодженому вигляду: логотип не повинен створювати штучні великі відступи навколо себе.

### Адаптивне розміщення логотипа hero
- На desktop логотип залишається у правій частині hero поруч із текстовим блоком.
- На tablet логотип має стояти праворуч від заголовка hero, орієнтовно в розкладці 60% ширини для тексту і 40% для логотипа.
- На tablet заголовок і логотип мають бути вирівняні одне відносно одного по вертикальному центру.
- На mobile логотип також має стояти праворуч від заголовка hero, а описовий текст, кнопки і блок довіри мають переходити нижче на повну ширину.
- На mobile і tablet логотип має бути достатньо помітним: після погодженого коригування його розмір збільшено приблизно на 20% відносно першого прийнятого варіанта.
- Заголовок hero на mobile має залишатися читабельним і не стискатися настільки, щоб втрачати ієрархію першого екрану.

### Швидкі переходи під hero
- Блок швидких переходів під hero має працювати як візуальна перемичка між hero і наступною секцією головної сторінки.
- Центр блоку швидких переходів має збігатися з межею між фоном hero і фоном наступної секції. Не допускається візуальне зміщення блоку вище або нижче цієї межі, особливо на tablet.
- Висота рядків швидких переходів має бути стабільною на кожному брейкпоінті, щоб половина фактичної висоти блоку була над межею, а половина - під нею.
- На desktop швидкі переходи показуються в один ряд із чотирьох елементів.
- На tablet швидкі переходи показуються у сітці 2×2 і мають залишатися відцентрованими відносно межі двох фонів.
- На mobile швидкі переходи показуються в одну колонку; весь вертикальний блок має залишатися прив’язаним до межі фонових секцій за тією ж логікою центрування.
- Поточні швидкі переходи головної сторінки: «Освітні програми», «Правила прийому», «Вступ», «Контакти».

- Прийняті тексти підблоків:
  - «Освіта»: «фахова підготовка з практичним змістом».
  - «Практика»: «практичні заняття, проєкти та підготовка до роботи за фахом».
  - «Підтримка»: «викладачі, ментори й студентська спільнота поруч».
  - «Спільнота»: «події, ініціативи та середовище для розвитку».
- На mobile заголовки підблоків мають вміщатися в межі екрана без горизонтального переповнення. Допускається зменшення шрифту або інша адаптивна підлаштовка в межах прийнятого візуального стилю.

### Блок «Останні новини» на головній сторінці
- Блок розміщується після інформаційної смуги під освітніми програмами.
- Основний заголовок блоку: «Життя коледжу».
- Опис під заголовком: «Реальні події, навчальні практики, творчі звіти та волонтерські ініціативи спільноти.»
- У верхній частині блоку має бути CTA-кнопка «Усі новини» з темно-синім фоном і білим текстом. На mobile кнопка займає всю ширину контейнера.
- На головній сторінці показуються **рівно три найновіші новини**, якщо у стрічці доступно три або більше матеріалів.
- «Найновіші» означає останні опубліковані матеріали у спільному джерелі новин, відсортовані від найновішого до найстарішого за датою публікації або погодженою редакційною датою.
- Картки головної сторінки використовують той самий дизайн, структуру і рендеринг, що й картки на сторінці «Новини».
- Картка новини містить фото, дату, тег циклової комісії, заголовок, короткий текст з оригінального матеріалу і посилання «Читати повністю».
- Теги не показуються поверх фото. Тег циклової комісії показується в мета-блоці картки під датою. Деталі - у розділі «Теги новин і фільтр за цикловою комісією». Інші мета-елементи (тип матеріалу, кнопки категорій, окремий мета-рядок) у картці не використовуються.
- Дата в картці береться з дати публікації WordPress-запису. Якщо для імпортованих матеріалів потрібно зберегти дату старого сайту, у WordPress допускається окреме редакційне поле видимої дати; тоді і сортування, і показ мають використовувати одну погоджену дату.
- Короткий текст у картці має братися зі змісту оригінальної статті або редакційного excerpt без переписування. Якщо текст довший за прийнятий ліміт картки, він скорочується з трьома крапками.
- Довгі заголовки і короткі тексти мають вкладатися в стабільну висоту картки: заголовок скорочується з трьома крапками, а короткий текст візуально центрується між заголовком і посиланням «Читати повністю».
- У статичній демоверсії новини для головної сторінки, сторінки «Новини» і сторінок окремих новин мають братися з одного локального джерела даних, а не дублюватися вручну в кількох HTML-файлах.

## Сторінка новин
- Пов’язані дизайн-рішення: `docs/decisions/DDR-010-news-archive-and-article-pages.md`, `docs/decisions/DDR-014-news-department-tags-and-filter.md`.
- Сторінка «Новини» показує загальну стрічку матеріалів із фільтром за цикловою комісією. Деталі фільтра - у розділі «Теги новин і фільтр за цикловою комісією».
- Картки на сторінці «Новини» мають виглядати рівно так само, як картки в блоці «Останні новини» на головній сторінці.
- Сторінка «Новини» показує матеріали зі спільного джерела, відсортовані від найновішої до найстарішої.
- На одній сторінці архіву має бути максимум **9 новин**.
- На desktop архів новин має формувати три ряди по три картки. На tablet допускається адаптація до двох колонок, на mobile - до однієї колонки.
- Перша сторінка архіву містить 9 найновіших новин. Друга сторінка містить наступні старіші новини, також максимум 9. Третя і наступні сторінки працюють за тією самою логікою.
- Якщо у стрічці 12 новин, у демоверсії перша сторінка показує 9 карток, а друга сторінка - 3 картки.
- Пагінація розміщується під сіткою новин. Вона має показувати доступні сторінки архіву і виділяти активну сторінку.
- У статичній демоверсії перша сторінка архіву доступна як `news.html`, а друга - як `news.html?page=2`. У WordPress потрібно використовувати стандартну пагінацію архіву або погоджені ЧПУ-посилання теми.
- Додавання нової новини має автоматично ставити її першою на сторінці 1 і зміщувати старіші матеріали далі по архіву.
- Hero сторінки новин не містить правого інформаційного або статистичного блоку. Сторінка «Новини» має solo-hero з breadcrumb, міткою розділу, заголовком і коротким описом.
- Додавання або оновлення новини у спільному джерелі має автоматично оновлювати і сторінку «Новини», і блок із трьома останніми новинами на головній сторінці.

## Сторінка окремої новини
- Пов’язані дизайн-рішення: `docs/decisions/DDR-010-news-archive-and-article-pages.md`, `docs/decisions/DDR-013-sdg-icons-on-news.md`, `docs/decisions/DDR-014-news-department-tags-and-filter.md`.
- Кожна новина має відкриватися на окремій сторінці сайту, а посилання «Читати повністю» у картках має вести на локальну сторінку поточного сайту, не на старий сайт.
- У статичній демоверсії окремі сторінки новин створені як HTML-файли з URL на кшталт `news-4875-...html`. У WordPress це має бути шаблон одиночного запису новини.
- Сторінка окремої новини використовує стандартні header і footer сайту.
- Верхня частина сторінки містить breadcrumb, мітку «Новина», заголовок новини та короткий вступний текст або excerpt.
- Основний блок матеріалу містить заголовок групи «Новини коледжу», дату новини, теги циклової комісії новини, повний текст матеріалу без змістових змін і всі зображення, які належать до оригінальної новини.
- На сторінці окремої новини показуються всі теги циклової комісії цієї новини (не лише перший), на відміну від компактного формату в картці.
- Під основним текстом матеріалу показується блок «Цілі сталого розвитку» з іконками пов’язаних цілей, якщо новина має такий зв’язок. Деталі - у розділі «Цілі сталого розвитку на сторінці новини».
- Дата на сторінці окремої новини має збігатися з датою в картці цієї новини. Базове джерело - дата публікації WordPress-запису; для імпортованих матеріалів допускається погоджене редакційне поле видимої дати.
- Якщо на сторінці окремої новини показуються теги циклових комісій, вони мають починатися з нового рядка після дати. Теги не повинні продовжувати той самий рядок, що й дата, і мають переноситися без горизонтального переповнення на mobile.
- Featured image використовується для картки новини, соціального прев’ю і, за потреби, як головне зображення матеріалу. Додаткові зображення зберігаються в контенті новини через медіабібліотеку WordPress.
- Правий або додатковий навігаційний блок сторінки містить посилання «Усі новини» та переходи до сусідніх матеріалів у межах того самого сортування: «Наступна новина» для новішої новини і «Попередня новина» для старішої.
- Якщо новина є найновішою або найстарішою у стрічці, відповідний сусідній перехід не показується або рендериться як неактивний елемент без хибного посилання.

## Теги новин і фільтр за цикловою комісією
- Пов’язане дизайн-рішення: `docs/decisions/DDR-014-news-department-tags-and-filter.md`.
- Кожна новина позначається тегом циклової комісії. У підписі тегу службова частина «Циклова комісія» опускається, підпис подається в називному відмінку.
- Прийняті теги циклових комісій:
  1. «Обліково-фінансові дисципліни»;
  2. «Економіка, торгівля та маркетинг»;
  3. «Соціально-гуманітарні та правознавчі дисципліни»;
  4. «Інформаційно-технічні та природничі дисципліни»;
  5. «Харчові технології, організація готельно-ресторанного бізнесу».
- У картці новини тег показується в мета-блоці: перший рядок - дата, наступний рядок - тег. Тег не показується поверх фото.
- Якщо назва тегу не вміщується в один рядок, вона скорочується з трьома крапками; повна назва доступна як підказка (`title`).
- Якщо новина має кілька тегів, у картці показується перший тег і компактна позначка «+N» із кількістю решти; повний перелік решти - у підказці. Уся група тегів лишається в один рядок.
- На сторінці окремої новини показуються всі теги новини.
- На сторінці «Новини» є ряд фільтрів за цикловою комісією: кнопка «Усі» і кнопка для кожної циклової комісії. Ряд фільтрів вирівнюється по лівому краю.
- Активний фільтр відображається в адресі як `?tag=…`. Фільтрування, пагінація і переходи назад/вперед у браузері відбуваються без повного перезавантаження сторінки і без стрибка до верху. Пагінація зберігає активний фільтр.
- Якщо за обраною цикловою комісією немає новин, показується зрозуміле повідомлення про відсутність матеріалів.
- Тег у картці й кнопка фільтра є робочими переходами до відфільтрованої стрічки.
- У статичній демоверсії тег новини зберігається у спільному джерелі (`js/news-data.js`, поле `tags`). На поточному етапі теги призначені довільно як заглушка і підлягають редакційному уточненню.
- У майбутній WordPress-реалізації циклова комісія новини має зберігатися як таксономія, а фільтр архіву - як серверний запит за цією таксономією.

## Цілі сталого розвитку на сторінці новини
- Пов’язане дизайн-рішення: `docs/decisions/DDR-013-sdg-icons-on-news.md`.
- На сторінці окремої новини під основним текстом матеріалу показується блок «Цілі сталого розвитку» з іконками пов’язаних цілей.
- Використовується офіційний набір 17 Цілей сталого розвитку. У демоверсії іконки зберігаються як окремі файли `assets/sdg/sdg-01.png` … `assets/sdg/sdg-17.png`.
- Новина може бути пов’язана з кількома цілями; усі пов’язані іконки показуються поряд із природним переносом рядків.
- Кожна іконка має локалізований опис (`alt`/`title`) формату «Ціль сталого розвитку N: назва».
- Блок ЦСР показується лише на сторінці окремої новини, а не в картках стрічки.
- У статичній демоверсії зв’язок новини з цілями зберігається у спільному джерелі (`js/news-data.js`, поле `sdgs`). На поточному етапі значення призначені довільно як заглушка і підлягають редакційному уточненню.
- У майбутній WordPress-реалізації зв’язок новини з цілями має зберігатися як таксономія або поле запису.

## Основні сторінки розділів
- Пов'язане дизайн-рішення: `docs/decisions/DDR-011-section-landing-pages.md`.
- Цей розділ описує тільки основні сторінки розділів: `college.html`, `admissions.html`, `students.html`, `alumni.html`, `science.html`, `library.html`.
- Дочірні HTML-сторінки, створені як заглушки або майбутні підсторінки, у цьому розділі не описуються як фінально затверджений контент. Для них потрібне окреме дослідження матеріалів чинного сайту, редакційне наповнення і рішення щодо статичної або динамічної моделі.
- Основні сторінки розділів мають бути **навігаційними hub-сторінками**: вони швидко показують структуру розділу, дають переходи до підсторінок і залишають на основній сторінці тільки ті змістові блоки, які окремо погоджені.
- На цих сторінках не використовувати декоративні праві hero-блоки з числами, роками, кількістю матеріалів або подібними підписами.

### Спільна структура hero для сторінок розділів
- Hero містить breadcrumb формату «Головна / Назва розділу».
- Після breadcrumb показуються мітка розділу, великий заголовок сторінки і короткий опис.
- Праворуч у hero на сторінках розділів показується блок «На цій сторінці».
- Блок «На цій сторінці» у hero є локальною навігацією по поточній сторінці. Його посилання ведуть до anchor-секцій на цій самій сторінці, а не до дочірніх HTML-сторінок.
- Назва блоку «На цій сторінці» використовується для того, щоб користувач розумів: це швидкий перехід усередині поточної сторінки.
- Колір заголовка блоку «На цій сторінці» у hero має збігатися з кольором заголовка бічної навігації.
- На широких екранах правий блок навігації вирівнюється по висоті з основним текстовим блоком hero, щоб не залишати зайву порожнечу знизу і вміщувати до п'яти локальних переходів.
- На tablet і mobile спеціального підняття правого блоку не потрібно: hero переходить у природну одноколонкову адаптивну розкладку.
- На mobile локальні посилання в hero показуються як компактні кнопки/рядки в межах ширини екрана, без горизонтального переповнення.

### Бічна навігація сторінкою
- На desktop/laptop сторінки розділів мають sticky-бічну навігацію з заголовком «На цій сторінці».
- Бічна навігація дублює той самий список anchor-переходів, що і блок у hero.
- Заголовок «На цій сторінці» у бічній навігації має бути вирівняний по горизонталі з текстом пунктів навігації.
- Бічна навігація показує тільки фактичні активні секції поточної сторінки. Не показувати посилання на секції, які прибрані з основної сторінки.
- На tablet і mobile бічна навігація приховується, щоб не дублювати верхній навігаційний блок і не займати корисну ширину контенту.

### Картки-посилання всередині секцій
- Секції, що відповідають групам розділу, містять grid із картками-посиланнями.
- Картка-посилання має містити назву, короткий опис і дію «Перейти».
- У картках не використовувати додаткові плашки-теги над назвою.
- Картки мають бути мінімалістичними: світлий фон, тонка межа, м'яка тінь, hover-стан із легким підняттям і акцентною межею.
- На desktop сітка карток показується у три колонки, на tablet - у дві, на mobile - в одну колонку.
- Довгі назви карток мають переноситися всередині картки без горизонтального скролу і без накладання на дію «Перейти».
- Назви секцій і карток мають збігатися з погодженими назвами в header-menu або з поточною очищеною структурою сторінки. Не змінювати їх редакційно без окремого погодження.

### Сторінка «Коледж»
- Hero:
  - мітка: «Коледж»;
  - заголовок: «Місце, де формується професійне майбутнє»;
  - опис: «Ми поєднуємо сильну академічну базу, практичне навчання та культуру взаємної підтримки.»
- Локальна навігація у hero і бічному блоці:
  1. «Загальна інформація»;
  2. «Основна інформація»;
  3. «Діяльність»;
  4. «Керівництво»;

- Footer є глобальним елементом сайту і має бути однаковим для всіх сторінок у межах кожної мовної версії.

## Типографіка
- Основною гарнітурою сайту є **Inter**.
- Базовий CSS-стек: `Inter, system-ui, sans-serif`.
- Inter має підключатися локально з файлів демоверсії: `assets/fonts/inter/InterVariable.woff2` і `assets/fonts/inter/InterVariable-Italic.woff2`.
- Файл ліцензії шрифту має зберігатися поруч зі шрифтовими файлами: `assets/fonts/inter/OFL.txt`.
- Для шрифту використовується `font-display: swap`, щоб текст залишався видимим під час завантаження font-файлу.
- `system-ui` і `sans-serif` є тільки fallback-варіантами, а не окремими погодженими гарнітурами сайту.
- Не додавати нові зовнішні шрифти, Google Fonts-підключення або платні/proprietary гарнітури без окремого дизайн-рішення.
- Типографічну ієрархію потрібно будувати через розміри, вагу, line-height, колір, відступи й адаптивні правила, а не через збільшення кількості гарнітур.

## Візуальний ритм і вертикальні відступи
- Перший екран головної сторінки не повинен мати надмірного порожнього простору між нижньою частиною навігаційного header і початком hero-контенту.
- На desktop текстовий блок hero і візуальний блок із логотипом мають починатися на одному рівні й мати компактний верхній відступ від навігації.
- На tablet і mobile заголовок hero має починатися компактно після header, а логотип має розміщуватися поруч із заголовком, не переходячи в окремий великий блок нижче.
- Прозорі поля в оригінальному файлі великого логотипа не повинні формувати візуальні відступи hero. Під час реалізації потрібно або обрізати видиму область, або використовувати підготовлений asset без зайвих прозорих полів.
- Верхній відступ контенту footer має бути компактним і однаковим за відчуттям на desktop, tablet і mobile. У поточній демоверсії використовується 33px зверху для сітки footer.
- Під час майбутніх змін головної сторінки не збільшувати ці відступи без окремого дизайн-рішення, навіть якщо змінюється текст або структура hero.

## Кандидат на затвердження: перемикач тем
### Статус
- **Потребує затвердження.** Функціонал перемикання тем описано для обговорення і тестування, але він ще не є фінально прийнятою вимогою до production або WordPress-реалізації.
- Перед включенням у фінальне технічне завдання потрібно окремо вирішити, чи лишається перемикач тем на сайті, чи це тільки інструмент демоперегляду.
- Також потрібно визначити фінальний набір тем: усі чотири поточні теми або скорочений набір, наприклад тільки «Основна» і «Нічний режим».

### Призначення
- Перемикач тем дає змогу переглядати сайт у кількох погоджуваних колірних напрямах без дублювання сторінок і без зміни структури контенту.
- Функція корисна для демонстрації дизайн-напрямів замовнику, перевірки нічного режиму і майбутнього рішення щодо доступності та комфорту читання.
- Тема має змінювати глобальні кольори сайту: фон, текст, muted-текст, межі, акценти, hero, картки, меню, кнопки, footer і службові панелі.
- Зміна теми не повинна змінювати тексти, ієрархію сторінок, порядок блоків, посилання або контентні дані.

### Поточний склад і порядок тем
- У випадаючому списку кнопки «Тема» теми показуються в такому порядку:
  1. «Основна» — базова світла синьо-бірюзова тема сайту.
  2. «Нічний режим» — темна тема для перегляду сайту на темному фоні.
  3. «Фіолетова» — альтернативна експериментальна палітра.
  4. «Бордова» — альтернативна експериментальна палітра.
- В англійській версії відповідні назви інтерфейсу: «Theme», «Primary», «Night mode», «Royal Purple», «Burgundy».
- Якщо фінально буде затверджено тільки основну тему і нічний режим, альтернативні теми «Фіолетова» і «Бордова» потрібно прибрати з інтерфейсу, але можна зберегти як внутрішні дизайн-нотатки або архів експериментів.

### Поведінка інтерфейсу
- На сторінках демоверсії є плаваюча кнопка «Тема» у правій нижній частині viewport.
- Клік або tap по кнопці відкриває компактну панель зі списком тем.
- Кожна тема має коротку назву і візуальний swatch, який показує характер палітри.
- Поточна активна тема має мати помітний активний стан.
- Вибір теми застосовується одразу, без перезавантаження сторінки.
- Обрана тема має зберігатися між сторінками і після оновлення сторінки в межах того самого браузера.
- Якщо збереженої теми немає або JavaScript недоступний, сайт має відкриватися в темі «Основна».
- Клік поза панеллю має закривати панель, не змінюючи поточну тему.

### Технічна модель
- Теми мають будуватися через глобальні дизайн-токени або CSS-змінні, а не через дублювання окремих стилів для кожної сторінки.
- Застосування теми має відбуватися на глобальному рівні документа, наприклад через атрибут `data-theme` на елементі `html`.
- Порожнє значення або відсутність `data-theme` відповідає темі «Основна».
- Стан вибраної теми у демоверсії може зберігатися в `localStorage`. У production потрібно окремо визначити, чи достатньо `localStorage`, чи потрібне серверне або cookie-збереження для узгодження з політикою приватності.
- Усі сторінки мають використовувати ті самі токени кольорів, щоб тема працювала глобально і не вимагала ручного налаштування кожної сторінки.
- Жодна сторінка не повинна мати жорстко зашиті світлі фони або темні тексти, які ламають нічний режим. Якщо окремий компонент має власний фон, він повинен мати відповідне theme-aware перевизначення.

### Доступність і адаптивність
- Кнопка перемикача має мати зрозумілий текстовий label: «Тема» / «Theme».
- Відкритий стан панелі має відображатися через `aria-expanded`.
- Варіанти тем мають бути доступні з клавіатури і мати зрозумілий активний стан.
- Колірні поєднання кожної теми потрібно перевіряти на читабельність основного тексту, muted-тексту, CTA, контурних кнопок, навігації, mega-menu, search dialog, карток новин, таблиць і footer.
- Нічний режим має забезпечувати достатній контраст без надмірно яскравих акцентів, щоб сайт залишався інституційним і зручним для читання.
- Перемикач тем не повинен перекривати основні кнопки, footer, кнопку «Повернутися вгору» або контент на mobile, tablet, laptop і wide viewport.
- Панель тем не повинна створювати горизонтальний скрол або обрізати довгі українські назви.

### Вимоги до майбутньої WordPress-реалізації, якщо функцію буде затверджено
- Перемикач тем має бути глобальним компонентом WordPress-теми, а не окремою ручною вставкою на кожній сторінці.
- Набір тем і їх порядок мають бути централізовано визначені в темі або в налаштуваннях сайту.
- Редактор WordPress не повинен випадково змінювати системні кольори окремої сторінки так, щоб це ламало глобальні теми.
- Усі Gutenberg blocks, custom blocks, template parts і глобальні компоненти мають використовувати спільні theme tokens.
- Якщо перемикач лишається публічною функцією, потрібно передбачити локалізацію назв тем українською та англійською.
- Якщо перемикач лишається лише інструментом для демо або погодження дизайну, його потрібно легко вимкнути перед production-публікацією.

### Відкриті питання
- Чи затверджується перемикач тем як публічна функція сайту?
- Чи потрібен фінальний набір із чотирьох тем, чи достатньо двох: «Основна» і «Нічний режим»?
- Чи має нічний режим бути доступним усім користувачам, чи тільки як внутрішній режим перегляду для команди?
- Де саме має розміщуватися перемикач у production-інтерфейсі, якщо плаваюча кнопка буде визнана надто демонстраційною?
- Який механізм збереження вибору теми затвердити для production: `localStorage`, cookie або серверне налаштування користувача?

## Доступність
- Кнопка мобільного меню має коректно оновлювати `aria-expanded`.
- Кнопка мобільного меню має мати зрозумілий `aria-label` для станів відкриття і закриття.
- Кнопки пунктів головної навігації з підменю мають коректно оновлювати `aria-expanded` і мати зв’язок із відповідним mega-menu через `aria-controls`.
- Для пунктів із підменю потрібно використовувати інтерактивний елемент типу button, а не посилання з перехопленим кліком.
- Мовний перемикач у мобільному меню має зберігати зрозумілу структуру посилань на мовні версії.
- Посилання на Instagram і Facebook мають мати зрозумілі `aria-label`.
- Локальні навігаційні блоки «На цій сторінці» мають мати зрозумілий `aria-label`, наприклад «Навігація сторінкою».
- Усі посилання локальної навігації на сторінках розділів мають вести до наявних у документі `id`. Не допускаються мертві anchor-посилання на прибрані секції.
- Вбудовані карти мають мати змістовний `title`, щоб користувач допоміжних технологій розумів призначення iframe.

## Вимоги до майбутньої WordPress-реалізації
- Header і footer мають бути реалізовані як глобальні компоненти теми або шаблону, а не як ручні копії на кожній сторінці.
- Тема WordPress має підтримувати локалізацію інтерфейсних рядків: назви меню, кнопки, labels, placeholders, службові повідомлення, 404, пошук, header і footer не повинні бути жорстко зашиті лише однією мовою.
- Тема WordPress має постачати Inter як локальний asset теми або погодженого asset-пакета. Не покладатися на встановлений у користувача Inter і не підключати Google Fonts/CDN без окремого рішення.
- Не вимагати від редактора дублювати кожну новину, оголошення, документ або медіафайл для англійської версії. Базовий редакційний сценарій: матеріал додається один раз українською і може відображатися також в англомовному інтерфейсі.
- Якщо в майбутньому буде використано multilingual-плагін, його потрібно налаштувати так, щоб повний англійський дубль динамічного контенту не був обов'язковою умовою публікації або показу матеріалу.
- Англійські версії створювати насамперед для погоджених статичних сторінок і глобальних елементів. Повний переклад динамічного контенту допускається тільки як додаткова редакційна можливість, а не як щоденна вимога.
- Медіафайли не повинні фізично дублюватися лише через мовну версію. Якщо потрібні перекладені alt-тексти або підписи, вони мають прив'язуватися до того самого медіаресурсу або керуватися без повторного завантаження файлу.
- Англомовна головна сторінка має мати перекладений статичний контент, але блок новин може підтягувати той самий український потік новин.
- Метадані соціальних прев’ю у WordPress мають формуватися централізовано для всіх сторінок, а не заповнюватися вручну в кожному шаблоні.
- WordPress-реалізація має забезпечити, щоб `og:image` і `twitter:image` для всіх сторінок за замовчуванням вказували на логотип коледжу або на інший окремо затверджений share-image asset.
- Якщо в майбутньому буде затверджено сторінкові описи для соціальних прев’ю, у WordPress потрібно передбачити кероване поле для такого опису з fallback на загальний опис сайту.
- Структура головного меню має відповідати розділу «Header і головна навігація» цього документа: назви пунктів, порядок, групи mega-menu і підпункти не змінювати без окремого погодження.
- У WordPress потрібно передбачити можливість керування URL пунктів меню, але зміна URL не повинна автоматично змінювати погоджені назви й ієрархію меню.
- У header має бути глобальна CTA-кнопка “Вступ” після пункту “Новини”.
- Для footer передбачити окрему редаговану групу швидких переходів під брендовим блоком. За замовчуванням вона містить: “Розклад занять”, “Освітні програми”, “Вступ”, “Правила прийому”, “Відділення”, “Циклові комісії”.
- Не додавати видимий заголовок до цієї групи швидких переходів без нового дизайн-рішення.
- Не додавати описовий текст під назву коледжу в брендовій колонці footer без нового дизайн-рішення.
- Не переносити соціальні мережі назад у колонку логотипа без окремого дизайн-рішення.
- Не повертати заголовок “Соціальні мережі” у footer, якщо не буде затверджено нову структуру footer.
- Реалізувати scroll lock для мобільного меню так, щоб фонова сторінка не рухалась під відкритим drawer.
- Реалізувати внутрішній скрол drawer для довгих розкритих меню.
- Якщо для мобільних підрозділів використовується accordion-анімація, вона має переходити у стабільний відкритий стан, який не блокує скрол drawer і не заважає touch-жестам на пунктах підменю.
- Реалізувати scroll lock і для tablet mega-menu: під відкритим меню фонова сторінка не рухається, а довгий вміст скролиться всередині самого mega-menu.
- Реалізація scroll lock не повинна глобально змінювати поведінку сторінки так, щоб sticky header, drawer або mega-menu зміщувалися поза видиму область після відкриття меню на проскроленій сторінці.
- Для tablet-меню передбачити адаптивну сітку груп. Довгі розділи не мають стискатися в надто багато колонок; групи повинні переноситися нижче.
- У WordPress-меню parent item з дочірніми пунктами має рендеритися як кнопка-перемикач. Parent item без дочірніх пунктів має рендеритися як звичайне посилання.
- Якщо parent item із дочірніми пунктами має власну сторінку-огляд, посилання на цю сторінку потрібно показувати всередині mega-menu, а не на самому перемикачі.
- Основні сторінки розділів «Коледж», «Абітурієнту», «Студенту», «Випускнику», «Наука» і «Бібліотека» у WordPress мають бути реалізовані як керовані hub-шаблони або page templates, а не як довгі ручні промосторінки.
- Для цих сторінок потрібно передбачити керовану локальну навігацію «На цій сторінці», яка автоматично або редакційно синхронізується з фактичними секціями сторінки.
- Картки-посилання в секціях сторінок розділів мають генеруватися зі структури відповідного розділу або з керованого repeatable field. Редактор може змінювати URL, але не повинен випадково змінювати затверджені назви без окремого рішення.
- Дочірні сторінки розділів потребують окремого етапу наповнення. Не переносити placeholder-тексти демоверсії у production як фінальний контент.
- Для секцій контактів на сторінках «Коледж» і «Абітурієнту» потрібно передбачити керовані поля для адреси, телефону, email, годин роботи, посилання Google Maps, соціальних мереж і embed-карти.
- У footer WordPress-реалізації мають бути глобальні колонки «бренд + швидкі переходи», «Навігація», «Контакти» з тією структурою і назвами, що описані в розділі «Footer».
- Не фіксувати висоту мобільного меню на весь екран, якщо контент коротший.
- Переконатися, що мобільний footer коректно центрує логотип відносно багаторядкової назви.
- Зберегти компактний вертикальний ритм першого екрана головної сторінки та footer, щоб після перенесення у WordPress не повернувся надмірний порожній простір.
- У WordPress-реалізації hero головної сторінки має зберегти погоджене обрізання великого логотипа. Можна використати CSS-обрізання або окремий підготовлений asset, але не можна повертати прозорі поля оригінального GIF у видиму область.
- У WordPress-реалізації швидкі переходи під hero мають залишатися окремим блоком-перемичкою на межі двох секцій, а не звичайною карткою всередині однієї секції.
- Блок «Про коледж» на головній сторінці має бути керованим з WordPress. Фото каруселі потрібно реалізувати як редаговану галерею або repeatable field: редактор має мати можливість додавати, видаляти, замінювати й міняти порядок фото без залучення розробника після початкового налаштування.
- Для кожного фото в WordPress потрібно передбачити редагований альтернативний текст.
- Кількість слайдів, точок і доступних кнопок каруселі має формуватися автоматично з фактичної кількості фото. Не зашивати кількість фото в шаблон так, щоб для додавання нового фото був потрібен розробник.
- Якщо слайд використовується як картинка-анонс події, редактор має мати змогу додати, прибрати або замінити такий анонс у WordPress без зміни коду шаблону. Альтернативний текст має передавати ключову інформацію з анонсу, особливо якщо на зображенні є текст.
- Текстовий контент блоку «Про коледж» також бажано зробити редагованим у WordPress: заголовок, короткий опис, три картки акцентів і CTA-посилання.
- Якщо у WordPress використовується Gutenberg, блок можна реалізувати як custom block або pattern з керованими полями. Якщо використовується ACF або подібний підхід, доречна структура: текстові поля для заголовка/опису/карток і gallery/repeater для фото.
- Блок «Основні розділи» на головній сторінці має зберігати погоджений порядок шести розділів і не додавати окрему картку «Публічна інформація». У WordPress цей блок можна реалізувати як керований pattern або custom block, але редакторські зміни не повинні випадково порушувати порядок: «Коледж», «Абітурієнту», «Студенту», «Випускнику», «Наука», «Бібліотека».
- Кнопка «Контакти» в блоці «Основні розділи» має вести до сторінки або секції контактів. Якщо контакти в майбутньому стануть окремою сторінкою, URL потрібно змінити централізовано без редагування кожної картки.
- Блок «Освітні програми» має бути окремим керованим блоком головної сторінки. У коротких картках не показувати тривалість навчання і форму навчання, якщо немає нового дизайн-рішення.
- Інформаційну смугу «Освіта / Практика / Підтримка / Спільнота» бажано реалізувати як керований набір із чотирьох коротких текстових підблоків, але зі збереженням погоджених назв і текстів за замовчуванням.
- Новини у WordPress бажано реалізувати як стандартні записи з окремою категорією «Новини» або як custom post type, якщо потрібна чітка редакційна модель для коледжу.
- Для новини потрібні: заголовок, основний текст, excerpt або автоматично сформований уривок, featured image з alt-текстом, дата публікації, URL повної сторінки матеріалу, а також усі додаткові зображення матеріалу в медіабібліотеці.
- Дата, яка показується в картках і на сторінці окремої новини, за замовчуванням береться з дати публікації WordPress-запису. Для імпорту зі старого сайту можна додати окреме редакційне поле видимої дати, але тоді це поле має послідовно використовуватися і для показу, і для сортування.
- На головній сторінці WordPress-шаблон має робити запит до трьох останніх опублікованих новин: `posts_per_page=3`, сортування за погодженою датою у спадному порядку.
- На сторінці «Новини» WordPress-шаблон має використовувати той самий template part картки й серверну пагінацію: `posts_per_page=9`, сторінка 1 показує найновіші 9 матеріалів, сторінка 2 - наступні старіші матеріали, і так далі.
- Шаблон окремої новини має показувати повний текст без змістових змін, усі зображення матеріалу, дату, посилання «Усі новини» та сусідню навігацію «Наступна новина» / «Попередня новина» на основі того самого сортування, що й архів.
- Якщо WordPress-шаблон показує теги або таксономії циклових комісій на single-сторінці новини, їх потрібно рендерити окремим блоком під датою.
- Не створювати окремий ручний список новин для головної сторінки у WordPress. Блок «Останні новини» має автоматично оновлюватися після публікації нової новини.
- У WordPress краще рендерити картки новин на стороні теми, а не покладатися тільки на клієнтський JavaScript, щоб зберегти SEO, доступність без JS і стабільне завантаження сторінки. JavaScript або REST API доречні як додатковий сценарій для фільтрів, пагінації чи повністю статичного фронтенду.

## Перевірка
Перед передачею реалізації потрібно виконати:
- Перевірку click/tap моделі головної навігації: пункт із підменю відкриває меню без переходу, повторний клік закриває меню, клік по іншому пункту перемикає меню.
- Перевірку, що пункт “Новини” або інший пункт без підменю лишається звичайним посиланням.
- Перевірку точних назв і порядку пунктів головного меню: «Коледж», «Абітурієнту», «Студенту», «Випускнику», «Наука», «Бібліотека», «Новини», «Вступ».
- Перевірку всіх груп mega-menu і підпунктів за списком у розділі «Header і головна навігація».
- Перевірку мобільного меню: відкрити меню, спробувати скролити фон, розкрити підрозділ і перевірити скрол усередині drawer.
- Регресійну перевірку довгого мобільного підрозділу: відкрити drawer, розкрити “Бібліотека” або інший довгий розділ і переконатися, що весь drawer можна прокрутити після завершення анімації.
- Перевірку tablet mega-menu: відкрити довгий розділ, наприклад “Бібліотека”, спробувати скролити фон і перевірити, що скролиться саме меню.
- Перевірку tablet-розкладки mega-menu: групи довгого розділу мають переноситися на кілька рядків без накладання тексту.
- Перевірку compact laptop viewport: довге mega-menu має не виходити за нижню межу екрана і має скролитися всередині власного контейнера.
- Перевірку після скролу сторінки: прокрутити сторінку вниз, відкрити мобільний drawer або mega-menu і переконатися, що відкритий шар меню видимий у поточному viewport.
- Перевірку, що header і footer структурно однакові на всіх основних сторінках у межах мовної версії.
- Перевірку службових кнопок header: на desktop/tablet кнопки пошуку, перемикання теми і перемикання мови мають однакову ширину й висоту.
- Перевірку типографіки: сайт має завантажувати локальний `InterVariable.woff2`, базовий стек має лишатися `Inter, system-ui, sans-serif`, а сторінки не повинні робити запитів до зовнішніх font-сервісів.
- Перевірку мовної моделі: англомовний інтерфейс має перекладати header, footer, пошук, CTA, службові елементи й погоджені статичні сторінки, але не вимагати англійського дубля для новин і динамічного контенту.
- Перевірку соціальних прев’ю: усі українські та англійські публічні сторінки мають `meta name="description"`, `og:title`, `og:description`, `og:url`, `og:image`, `twitter:card`, `twitter:title`, `twitter:description` і `twitter:image`; зображення прев’ю має бути доступне за публічним абсолютним URL.
- Перевірку редакційного сценарію WordPress: новина або оголошення, додані один раз українською, мають бути доступні для показу в українській версії і в англомовних блоках новин без повторного створення матеріалу.
- Перевірку англомовної головної сторінки: статичні секції перекладені англійською, а блок новин може показувати українські заголовки та матеріали.
- Перевірку footer на ширинах 375px, 390px, 428px і 430px.
- Перевірку footer на desktop і tablet: швидкі переходи мають бути під брендовим блоком без заголовка, колонка “Навігація” має виглядати збалансовано між брендовим блоком і контактами.
- Перевірку точних назв і порядку швидких переходів footer: «Розклад занять», «Освітні програми», «Вступ», «Правила прийому», «Відділення», «Циклові комісії».
- Перевірку, що footer однаковий на всіх сторінках у межах мовної версії.
- Перевірку, що швидкі переходи footer не створюють горизонтального переповнення і не обрізають українські назви.
- Перевірку hero головної сторінки на desktop: офіційний логотип має бути обрізаний без зайвих прозорих полів і не створювати надмірного нижнього або верхнього відступу.
- Перевірку hero головної сторінки на tablet: логотип має стояти праворуч від заголовка, бути вертикально збалансованим із ним, а quick-links мають бути точно на межі hero і наступної секції.
- Перевірку hero головної сторінки на mobile: логотип має стояти праворуч від заголовка, не перекривати текст, не створювати горизонтального скролу, а quick-links мають залишатися центрованими на межі секцій.
- Перевірку блоку «Про коледж» на головній сторінці: секція має бути між quick-links і «Основними розділами», карусель має містити фото, точки мають відповідати кількості фото, кнопки «попереднє/наступне фото» мають перемикати слайд, CTA має вести на сторінку про коледж.
- Перевірку блоку «Основні розділи» на головній сторінці: є шість карток у погодженому порядку, немає номерів 01-06, немає окремої картки «Публічна інформація», кнопка «Контакти» веде на контакти, на mobile кнопка стає під описом і займає всю ширину контейнера.

- Перевірку інформаційної смуги під освітніми програмами: чотири підблоки мають погоджені тексти, а заголовки «Практика» і «Спільнота» не виходять за межі екрана на mobile.
- Перевірку блоку «Останні новини» на головній сторінці: показуються рівно три найновіші картки зі спільного джерела, CTA «Усі новини» має темно-синій фон, на mobile CTA займає всю ширину, у картках є дата, але немає тегів, фільтрів або додаткового мета-рядка.

- Перевірку сторінки окремої новини: посилання «Читати повністю» веде на локальну сторінку поточного сайту, показано повний текст і зображення матеріалу, дата збігається з карткою, теги починаються окремим рядком після дати, а навігація «Усі новини», «Наступна новина» і «Попередня новина» працює без переходів на старий сайт.
- Перевірку WordPress-сценарію: після публікації нової новини вона автоматично стає першою на сторінці «Новини» і входить до блоку з трьома останніми новинами на головній сторінці без ручного редагування головної сторінки.
- Перевірку основних сторінок розділів: hero-навігація «На цій сторінці», бічна навігація і фактичні секції сторінки мають містити однаковий список активних розділів.
- Перевірку сторінок «Коледж», «Абітурієнту», «Студенту», «Випускнику», «Наука», «Бібліотека»: прибрані довгі секції не повинні повертатися на основні сторінки без окремого рішення.
- Перевірку карток-посилань на основних сторінках розділів: усі картки мають назву, короткий опис, посилання «Перейти» і hover-стан; довгі назви не мають накладатися на кнопку або виходити за межі картки.
- Перевірку desktop/laptop сторінок розділів: sticky-бічна навігація має бути видимою, а її заголовок «На цій сторінці» має бути вирівняний із текстом пунктів.

- Для нічного режиму окремо перевірити mobile, tablet, laptop і wide viewport: текст, контурні кнопки, search dialog, mega-menu, швидкі переходи, footer і плаваючі кнопки мають залишатися контрастними та не накладатися одне на одного.

## Відомі обмеження демоверсії
- Поточна реалізація є статичною демоверсією, а не WordPress-темою.
- Для сумісності з GitHub Pages готові HTML-файли містять розмітку header і footer у кожній сторінці. Це прийнятно для демоверсії лише за умови синхронізації з одного локального джерела перед публікацією.
- У статичній демоверсії новини беруться з локального файлу `js/news-data.js`, а повні матеріали існують як окремі HTML-сторінки. Це прибирає дублювання між головною сторінкою, сторінкою «Новини» і сторінками окремих новин, але не замінює повноцінну WordPress-адмінку: у production новини мають додаватися через редактор WordPress.
- Дочірні сторінки розділів, створені у демоверсії як заглушки, не є фінально погодженим контентом. Вони потребують окремого етапу дослідження чинного сайту, редакційного наповнення і технічного рішення щодо статичної або динамічної моделі.
- Перемикач тем у поточній демоверсії є кандидатом на затвердження. Його не слід автоматично вважати обов'язковою production-вимогою без окремого рішення щодо фінального набору тем і місця цього елемента в інтерфейсі.
- Частина посилань може бути демонстраційною і потребує заміни на реальні URL під час production-реалізації.
- Cache-buster у HTML використовується для демоверсії; у production-середовищі механізм версіонування ресурсів має визначатися збіркою або WordPress-темою.
